4. Flujo funcional y endpoints mínimos

La API deberá implementar el flujo completo de creación, publicación, inscripción y gestión de eventos académicos. Todos los endpoints indicados son obligatorios y utilizarán como ruta base /api. Los estudiantes podrán agregar otros endpoints cuando sean necesarios para completar su solución.

4.1. Flujo principal del sistema

  1. Un participante se registra mediante correo y contraseña. La cuenta se crea con el rol PARTICIPANT y estado ACTIVE.
  2. El administrador consulta los usuarios registrados y puede asignar el rol ORGANIZER, bloquear o reactivar una cuenta.
  3. Un organizador crea un evento en estado DRAFT, registra sus sesiones y posteriormente lo publica.
  4. Los usuarios pueden consultar los eventos publicados sin autenticarse, aplicando búsqueda, filtros, paginación y ordenamiento.
  5. Un participante autenticado solicita su inscripción en un evento. La inscripción se registra inicialmente con estado PENDING.
  6. El organizador propietario del evento revisa la solicitud y puede confirmarla o rechazarla.
  7. Cuando una inscripción es confirmada, el sistema deberá reducir los cupos disponibles dentro de una transacción.
  8. Si una inscripción confirmada es cancelada, el sistema deberá liberar nuevamente el cupo.
  9. El participante podrá consultar sus inscripciones y descargar su comprobante cuando la inscripción se encuentre confirmada.
  10. El organizador y el administrador podrán consultar estadísticas y descargar reportes de los eventos.

4.2. Estados permitidos

Recurso Estados Transiciones permitidas
Usuario ACTIVE, BLOCKED ACTIVE → BLOCKED
BLOCKED → ACTIVE
Evento DRAFT, PUBLISHED, FINISHED, CANCELLED DRAFT → PUBLISHED
PUBLISHED → FINISHED
DRAFT → CANCELLED
PUBLISHED → CANCELLED
Inscripción PENDING, CONFIRMED, REJECTED, CANCELLED PENDING → CONFIRMED
PENDING → REJECTED
PENDING → CANCELLED
CONFIRMED → CANCELLED

4.3. Módulo de autenticación

Permitirá registrar participantes, iniciar y cerrar sesión, renovar el access token y consultar la información del usuario autenticado.

Método Endpoint Acceso Función
POST /api/auth/register Público Registrar un participante.
POST /api/auth/login Público Autenticar y devolver access token y refresh token.
POST /api/auth/refresh Refresh token Generar un nuevo access token.
POST /api/auth/logout Autenticado
GET /api/auth/me Autenticado Consultar el perfil autenticado.

4.4. Módulo de usuarios

Permitirá al administrador consultar usuarios, asignar roles y cambiar el estado de las cuentas. Un participante no podrá asignarse roles ni modificar el estado de otros usuarios.

Método Endpoint Acceso Función
GET /api/users ADMIN Listar usuarios con paginación y filtros.
GET /api/users/{id} ADMIN Consultar un usuario.
PUT /api/users/{id}/roles ADMIN Asignar o retirar roles.
PATCH /api/users/{id}/status ADMIN Bloquear o reactivar una cuenta.

4.5. Módulo de categorías

Permitirá administrar las categorías utilizadas para clasificar los eventos. No se permitirán nombres duplicados.

Método Endpoint Acceso Función
POST /api/categories ADMIN Crear una categoría.
GET /api/categories Público Listar categorías activas.
GET /api/categories/{id} Público Consultar una categoría.
PUT /api/categories/{id} ADMIN Actualizar una categoría.
DELETE /api/categories/{id} ADMIN Realizar eliminación lógica.

4.6. Módulo de eventos

Permitirá a los organizadores gestionar sus propios eventos. Los usuarios públicos únicamente podrán consultar eventos con estado PUBLISHED.

Método Endpoint Acceso Función
POST /api/events ORGANIZER Crear un evento en estado DRAFT.
GET /api/events Público Listar eventos publicados con filtros y paginación.
GET /api/events/{id} Público Consultar el detalle de un evento publicado.
GET /api/events/mine ORGANIZER Consultar los eventos del organizador autenticado.

GET /api/events?modality=VIRTUAL

Método Endpoint Acceso Función

GET /api/events/{eventId}/sessions Público Listar las sesiones de un evento publicado.

DELETE /api/events/{eventId}/sessions/{sessionId} Propietario o ADMIN Eliminar una sesión.

4.8. Módulo de inscripciones

Permitirá completar el flujo entre participantes y organizadores. Cada cambio de estado deberá registrarse en el historial.

Método Endpoint Acceso Función
POST /api/events/{eventId}/registrations PARTICIPANT Solicitar una inscripción.
GET /api/registrations/me PARTICIPANT Consultar las inscripciones propias.
GET /api/events/{eventId}/registrations Propietario o ADMIN Listar inscritos con filtros y paginación.
GET /api/registrations/{id} Propietario, participante o ADMIN Consultar una inscripción.
PATCH /api/registrations/{id}/status Propietario o ADMIN Confirmar o rechazar una inscripción.
DELETE /api/registrations/{id} Participante propietario Cancelar la inscripción propia.

4.9. Módulo de auditoría

El sistema deberá registrar automáticamente las operaciones críticas realizadas por los usuarios, como inicios de sesión, cambios de roles, cambios de estado, eliminaciones lógicas y cualquier otra operación que afecte la integridad de los datos.

Los registros deberán generarse internamente desde la capa de servicio. No se deberán crear endpoints para insertar, modificar o eliminar registros de auditoría.

4.9.1. Flujo resumido de auditoría

Como ejemplo, el administrador con identificador 15 asigna el rol ORGANIZER al usuario con identificador 28.

Solicitud realizada:

PUT /api/users/28/roles
Authorization: Bearer JWT
Content-Type: application/json

Contenido de la solicitud:

{
  "roles": [
    "PARTICIPANT",
    "ORGANIZER"
  ]
}

Flujo ejecutado por el sistema:

  1. Spring Security identifica al administrador autenticado.
  2. El servicio consulta al usuario con identificador 28.
  3. Verifica que el usuario exista y que los roles sean válidos.
  4. Obtiene los roles actuales del usuario.
  5. Actualiza los roles dentro de una transacción.
  6. Registra automáticamente la operación en la tabla audit_logs.
  7. Devuelve la información actualizada del usuario.

Respuesta esperada:

HTTP/1.1 200 OK
Content-Type: application/json
{
  "id": 28,
  "email": "participant@correo.com",
  "status": "ACTIVE",
  "roles": [
    "PARTICIPANT",
    "ORGANIZER"
  ]
}

Registro de auditoría generado:

{
  "actorId": 15,
  "action": "USER_ROLES_UPDATED",
  "resourceType": "USER",
  "resourceId": 28,
  "previousValue": {
    "roles": [
      "PARTICIPANT"
    ]
  },
  "newValue": {
    "roles": [
      "PARTICIPANT",
      "ORGANIZER"
    ]
  },
  "result": "SUCCESS",
  "ipAddress": "192.168.1.20",
  "httpMethod": "PUT",
  "endpoint": "/api/users/28/roles",
  "createdAt": "2026-07-15T10:30:00-05:00"
}

La información almacenada permitirá identificar que:

El administrador 15 asignó el rol ORGANIZER al usuario 28.
Fecha: 2026-07-15 10:30
IP: 192.168.1.20
Resultado: SUCCESS

4.9.2. Estructura de la tabla audit_logs

Columna Tipo SQL Restricción Descripción
id BIGSERIAL PRIMARY KEY Identificador del registro de auditoría.
actor_id BIGINT FOREIGN KEY, NULL Usuario que ejecutó la operación. Puede ser nulo en acciones no autenticadas.
action VARCHAR(60) NOT NULL Acción realizada, por ejemplo USER_ROLES_UPDATED.
resource_type VARCHAR(50) NOT NULL Tipo de recurso afectado, como USER o EVENT.
resource_id BIGINT NULL Identificador del recurso afectado.
previous_value JSONB NULL Valores existentes antes de la operación.
new_value JSONB NULL Valores almacenados después de la operación.
result VARCHAR(20) NOT NULL Resultado de la operación: SUCCESS o FAILED.
ip_address VARCHAR(45) NULL Dirección IPv4 o IPv6 desde la que se realizó la solicitud.
http_method VARCHAR(10) NULL Método HTTP utilizado.
endpoint VARCHAR(255) NULL Endpoint sobre el que se realizó la operación.
correlation_id VARCHAR(36) NULL Identificador utilizado para relacionar la operación con los logs técnicos.
created_at TIMESTAMP WITH TIME ZONE NOT NULL Fecha y hora en que ocurrió la operación.

La auditoría no deberá almacenar contraseñas, access tokens, refresh tokens ni otra información sensible.

4.9.3. Endpoints sobre los que se aplicará la auditoría

La auditoría deberá aplicarse obligatoriamente sobre los siguientes tres endpoints del sistema. Cada vez que uno de ellos sea ejecutado, se deberá generar automáticamente un registro en la tabla audit_logs.

Módulo Método Endpoint Acciones de auditoría Información que se deberá registrar
Autenticación POST /api/auth/login LOGIN_SUCCESS
LOGIN_FAILED 		Usuario o correo utilizado, resultado, dirección IP, fecha, método HTTP y endpoint.
Usuarios PUT /api/users/{id}/roles USER_ROLES_UPDATED Administrador que ejecutó la operación, usuario afectado, roles anteriores, roles nuevos, resultado, dirección IP y fecha.
Inscripciones PATCH /api/registrations/{id}/status REGISTRATION_STATUS_CHANGED Usuario que ejecutó la operación, inscripción afectada, estado anterior, estado nuevo, resultado, dirección IP y fecha.

Los registros deberán generarse desde la capa de servicio después de ejecutar la operación. En los intentos fallidos de inicio de sesión, la auditoría deberá registrarse aunque el usuario no se encuentre autenticado.

No se requiere desarrollar un controlador ni endpoints propios para crear, modificar o eliminar registros de auditoría. La información será generada internamente y podrá verificarse directamente en la tabla audit_logs.
