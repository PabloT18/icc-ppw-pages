OBJETIVO

Evaluar la capacidad para identificar la capa correcta de una aplicación Spring Boot, modificar contratos JSON, construir consultas JPA, aplicar seguridad por roles, manejar excepciones centralizadas y desarrollar de extremo a extremo un módulo REST basado en una sola entidad.
INSTRUCCIONES:
  1. Clone únicamente la variante compatible con el JDK instalado: Java 17, Java 21 o Java 25.
  2. Ejecute el proyecto antes de modificarlo y confirme que inicia correctamente.
  3. No modifique schema.sql ni data.sql.
  4. No cambie nombres de rutas, campos JSON, mensajes, códigos internos ni códigos HTTP definidos.
  5. Toda regla de negocio debe permanecer en la capa de servicio.
  6. No se permite retornar entidades JPA desde controladores.
  7. La entrega será el enlace del repositorio remoto dentro del tiempo indicado.

CONTEXTO DEL SISTEMA

La universidad dispone de una API para administrar usuarios, campus, laboratorios, equipamiento e insumos. El proyecto entregado ya contiene los módulos de seguridad, usuarios y laboratorios. La base H2 se crea y carga automáticamente desde src/main/resources/schema.sql y data.sql.

La base contiene las tablas users, roles, user_roles, campuses, laboratories, equipment, laboratory_equipment, supplies y refresh_tokens.

Alcance: se evaluarán modificaciones concretas sobre módulos existentes y el desarrollo completo de un solo dominio nuevo: supplies. Este módulo utiliza exclusivamente la tabla supplies.

ACTIVIDADES POR DESARROLLAR

R1 · 0,50 puntos

Configuración de ejecución y JWT

Modificar el archivo necesario para que la API se ejecute en el puerto 8091. Cambiar la expiración del access token a 900000 milisegundos. Mantener el context path /api.

Capa afectada: configuración.

R2 · 1,00 punto

Usuarios elegibles y nuevo contrato de respuesta

Modificar GET /api/users/eligible para retornar únicamente usuarios con age >= 18, active = true y deleted = false, ordenados por fullName ascendente.

El JSON debe contener exactamente: id, fullName, email, age, active y roles. El campo name no debe existir.

El metodo JPA:

  • findByAgeGreaterThanEqual
  • AndActiveTrue
  • AndDeletedFalse
  • OrderByFullNameAsc

Capas afectadas: DTO, mapper, repository y service.

R3 · 1,00 punto

Consulta de laboratorios disponibles

Corregir GET /api/laboratories/available?campusId={id}&minCapacity={value}. Debe retornar laboratorios con campus coincidente, capacidad mayor o igual al mínimo, laboratorio activo, campus activo y ambos no eliminados. Ordenar por capacidad descendente.

El método del repository debe ser:

findByCampus_IdAndCapacityGreaterThanEqualAndActiveTrueAndDeletedFalseAndCampus_ActiveTrueAndCampus_DeletedFalseOrderByCapacityDesc

findByCampus_IdAndCapacityGreaterThanEqualAndActiveTrueAndDeletedFalseAndCampus_ActiveTrueAndCampus_DeletedFalseOrderByCapacityDesc

Retorna List<LaboratoryEntity> y recibe Long campusId, Integer minCapacity.

Capas afectadas: repository y service.

R5 · 0,50 puntos

Autorización

Aplicar estas reglas después de implementar los endpoints:

  • /users/eligible: únicamente ADMIN.
  • /laboratories/available: cualquier usuario autenticado.
  • GET /supplies/low-stock: cualquier usuario autenticado.
  • Crear, actualizar cantidad y eliminar insumos: ADMIN o COORDINATOR.

Capa afectada: controllers.

R6 · 2,00 puntos

Crear un insumo

Desarrollar el módulo supplies completo: entity, DTO, mapper, repository, service, implementación, controller y excepción. El módulo debe inyectar únicamente SupplyRepository. 

POST /api/supplies
{
  "name": "Adaptador USB-C",
  "description": "Adaptador para proyectores",
  "quantity": 7,
  "minimumStock": 3,
  "unitPrice": 12.50
}

Reglas:

Para que se pueda crear un registo, el nombre debe ser obligatorio y único entre registros no eliminados (existsByNameIgnoreCaseAndDeletedFalse). Si el nombre ya existe, responder 409 CONFLICT con código interno SUPPLY_CONFLICT.

La SupplyConflictException es el Requerimiento 10, sera una exception que extienda de ApplicationException. En el constructor super este: super(HttpStatus.CONFLICT, "SUPPLY_CONFLICT", message);

La cantidad y stock mínimo mayores o iguales a cero; precio mayor que cero. Crear con active = true y responder 201 CREATED.

El metodo del servicio sera SupplyResponseDto create(CreateSupplyDto dto);

Respuesta exacta: id, name, description, quantity, minimumStock, unitPrice y active.

R7 · 1,00 punto

Listar insumos con cantidad menor al límite

Implementar GET /api/supplies/low-stock?maxQuantity=10. Retornar únicamente registros con active = true, deleted = false y quantity < maxQuantity. Ordenar por quantity ascendente. Responder siempre 200 OK, incluso con lista vacía.

El repository debe utilizar un método derivado de Spring Data JPA y consultar únicamente SupplyEntity.

Para esta consulta usar el siguiente metodo:

  • Desde el reposotio con consulta JPA findByActiveTrue AndDeletedFalse AndQuantityLessThan OrderByQuantityAsc
  • Con consulta SQL
  • Y traer todos los elemntos y filtrar en codigo JAVA

Escoger la mejor implementacion

R8 · 1,00 punto

Actualizar la cantidad

Implementar PATCH /api/supplies/{id}/quantity.

{
  "quantity": 25
}

Actualizar exclusivamente quantity. El valor debe ser mayor o igual a cero. Si el registro no existe o está eliminado, responder 404 con código SUPPLY_NOT_FOUND. Retornar el DTO actualizado con 200 OK.

R9 · 1,00 punto

Eliminar lógicamente un insumo

Implementar DELETE /api/supplies/{id}. La eliminación debe asignar deleted = true y active = false. Solo se puede eliminar un insumo cuando su cantidad sea 0. Si se elimina correctamente, responder 204 NO CONTENT.

Si la cantidad no es cero lanzar una exception SupplyConflictException("Supply cannot be deleted while quantity is greater than zero");

R10 · 1,50 puntos

Excepción específica del módulo

Crear SupplyConflictException extendiendo ApplicationException. Debe responder 409 CONFLICT con código interno SUPPLY_CONFLICT.

Utilizar esta excepción en dos casos:

  • Intentar crear un nombre ya registrado entre insumos no eliminados.
  • Intentar eliminar un insumo cuya cantidad sea mayor que cero.
HTTP 409
{
  "code": "SUPPLY_CONFLICT",
  "message": "Supply name already registered"
}
